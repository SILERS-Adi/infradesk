# Plan wdrożenia RLS — InfraDesk v2 (2026-04-24)

**Cel:** Druga linia obrony izolacji workspace na poziomie PostgreSQL. Pierwsza linia (app middleware) zostaje.

**Zasada pracy:** każdy etap niezależny, z rollback, z weryfikacją. Żaden etap nie łamie produkcji — jeśli cokolwiek pójdzie źle, rollback w <30s.

## Stan wyjściowy (zweryfikowany 2026-04-24)

### Co JUŻ działa po stronie DB
- **49 z 57 tabel** ma `rowsecurity=t` **i** `relforcerowsecurity=t` (FORCE RLS jest już włączone!)
- **50 policies** w `pg_policy`, każda typu `ALL` (pokrywa SELECT/INSERT/UPDATE/DELETE w jednym)
- **3 funkcje helper** zdefiniowane: `app_is_super_admin()`, `app_current_workspace()`, `app_current_user()` — wszystkie korzystają z `current_setting('app.*', true)` z session config
- Policy pattern solidny:
  - **Direct workspaceId** (~34 tabel): `app_is_super_admin() OR "workspaceId" = app_current_workspace()`
  - **MSP cross-read** (Device, Location, MonitoringAlert, DownloadFile): rozszerzone USING z `WorkspaceRelation` subquery
  - **Parent-child** (BackupHistory, CredentialViewLog, InvoiceItem, TicketComment, TicketEvent, AccessGrant, etc.): EXISTS z tabeli nadrzędnej
  - **User-self** (NotificationSubscription): `"userId" = app_current_user()`
  - **Workspace** sam: 2 policies (modify + select)

### Co NIE działa — faktyczne dziury
1. 🚨 **`ALTER USER infradesk_v2 WITH BYPASSRLS`** → każde zapytanie omija polityki
2. 🚨 **`src/middleware/rls.ts::applyRlsContext`** zdefiniowany, ale **nigdzie nie podpięty** — `app.*` session vars nigdy nie są ustawiane
3. ⚠️ **3 tabele bez RLS** mimo `workspaceId`: `EmailMessage`, `UserEmailAccount`, `PermissionSchema`
4. ⚠️ `PermissionSchema` — global lookup table (schemas ról), może faktycznie nie potrzebować RLS — do decyzji

6. **`modules/auth-google/google.service.ts`** — OAuth flow pre-workspace → bg
7. **`modules/agents-ws/agents-ws.server.ts`** — WebSocket server, agent lookup by token, heartbeat updates. Pre-session workspace. → bg (po lookup można zrobić `SET LOCAL app.current_workspace` per-connection, ale prościej bg)

### 🟩 Etap 1 — Dodać RLS na 3 brakujące tabele
**Nie zmienia niczego runtime** (user wciąż BYPASSRLS), ale przygotowuje schemat.

**Działania:**
```sql
BEGIN;

-- EmailMessage: standard workspace pattern
ALTER TABLE "EmailMessage" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "EmailMessage" FORCE ROW LEVEL SECURITY;
CREATE POLICY "EmailMessage_tenant" ON "EmailMessage" FOR ALL
  USING (app_is_super_admin() OR "workspaceId" = app_current_workspace())
  WITH CHECK (app_is_super_admin() OR "workspaceId" = app_current_workspace());

-- UserEmailAccount: standard workspace pattern (userId jest dodatkowym filtrem w app)
ALTER TABLE "UserEmailAccount" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "UserEmailAccount" FORCE ROW LEVEL SECURITY;
CREATE POLICY "UserEmailAccount_tenant" ON "UserEmailAccount" FOR ALL
  USING (app_is_super_admin() OR "workspaceId" = app_current_workspace())
  WITH CHECK (app_is_super_admin() OR "workspaceId" = app_current_workspace());

-- PermissionSchema: jeśli ma workspaceId, to workspace-scoped. Jeśli NULL — globalny (role schemas)
-- Decyzja: allow anonymous SELECT na NULL (globalne), reszta workspace-scoped
ALTER TABLE "PermissionSchema" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "PermissionSchema" FORCE ROW LEVEL SECURITY;
CREATE POLICY "PermissionSchema_global_read" ON "PermissionSchema" FOR SELECT
  USING ("workspaceId" IS NULL OR app_is_super_admin() OR "workspaceId" = app_current_workspace());
CREATE POLICY "PermissionSchema_tenant_write" ON "PermissionSchema" FOR ALL
  USING (app_is_super_admin() OR "workspaceId" = app_current_workspace())
  WITH CHECK (app_is_super_admin() OR "workspaceId" = app_current_workspace());

COMMIT;
```

**Weryfikacja:**
- `SELECT tablename, rowsecurity, relforcerowsecurity FROM pg_tables WHERE tablename IN (...)` → wszystkie `t`
- `SELECT count(*) FROM pg_policy WHERE polrelid IN (...)` → 4 policies

**Rollback:**
```sql

DROP POLICY "EmailMessage_tenant" ON "EmailMessage";
-- analogicznie dla pozostałych
```

**Ryzyko:** zerowe — user wciąż BYPASSRLS, nic się w runtime nie zmieni.

### 🟨 Etap 4 — Podpiąć `applyRlsContext` middleware
**Działania:**

1. `src/middleware/rls.ts` — zmiana: **wpisać z `prismaBg` zamiast `prisma`** (bo ten middleware ma być wywołany ZANIM user kontekst jest ustawiony, ale kontekst musi być ustawiony PRZED użytkowym queryem regular clienta):
```typescript
// WAIT — to nie zadziała. applyRlsContext używa prisma.$executeRawUnsafe
// które ustawia session variable. Session variable jest per-connection.
// W Prisma pool connection może być re-użyte, więc SET LOCAL działa tylko w transakcji.
// Alternatywa: SET (bez LOCAL) na connection, ale ryzyko leakage między requestami.
// 
// Poprawne podejście: każdy HTTP request musi być w transakcji która ustawia
// SET LOCAL ... na początku. Prisma ma $transaction API.
// 
// ALBO: używać connection pool opts że każda kwerenda otwiera nową transakcję.
// 
// ALBO: użyć Prisma middleware/extension żeby każde query było w transakcji
// z SET LOCAL.
```

**⚠️ KRYTYCZNE ODKRYCIE TUTAJ:** obecny kod `applyRlsContext` jest **błędny**:
- `prisma.$executeRawUnsafe('SELECT set_config(...)', ...)` ustawia session var, ale **connection pool może użyć innego połączenia dla kolejnego query!** Więc ten set nie gwarantuje że następny query w tym samym request'cie zobaczy kontekst.
- Właściwy wzorzec: każde logiczne "unit of work" musi być w `prisma.$transaction()` z SET LOCAL na początku

**Opcje naprawy:**
- **Opcja 4A:** Prisma client extension ($extends) który opakowuje każde query w transakcję z SET LOCAL. **Czysto ale wymaga ~50-100 linii kodu i testów.**
- **Opcja 4B:** Użyć `SET` (nie `SET LOCAL`) + reset na końcu requesta. **Ryzyko session var leakage między requestami jeśli res nie wyreseta** (np. exception przed cleanup).
- **Opcja 4C:** Każdy router handler sam otacza logikę w `prisma.$transaction(async (tx) => { await tx.$executeRaw`SET LOCAL ...`; ... })`. **Najbezpieczniejsze ale inwazyjne** — trzeba zmienić każdy handler.

**Rekomendacja:** **Opcja 4A** — Prisma extension. Jedna definicja, transparentna dla kodu.

**Działania (opcja A):**
1. Stworzyć `src/lib/prisma.ts` z extension:
```typescript
import { PrismaClient } from '@prisma/client';

const base = new PrismaClient({ log: ['warn', 'error'] });

// Extension: every query runs in an implicit transaction that sets RLS session vars.
// This guarantees every statement sees app.current_workspace on the same connection.
export const prisma = base.$extends({
  query: {
    $allOperations: async ({ args, query, model, operation }) => {
      const ctx = getCurrentRequestContext(); // from AsyncLocalStorage
      if (!ctx) return query(args); // no context = admin/bg-like, but this client is regular — should never happen in RLS mode
      
      return base.$transaction(async (tx) => {
        await tx.$executeRawUnsafe(
          `SELECT set_config('app.current_workspace', $1, true),
                  set_config('app.current_user', $2, true),
                  set_config('app.is_super_admin', $3, true)`,
          ctx.workspaceId ?? '',
          ctx.userId,
          ctx.isSuperAdmin ? '1' : '0',
        );
        // Forward the original query to tx
        return (tx as any)[model!][operation](args);
      });
    },
  },
});
```

2. `AsyncLocalStorage` z Node.js dla request context:
```typescript
// src/lib/requestContext.ts
import { AsyncLocalStorage } from 'async_hooks';
export const requestContextStore = new AsyncLocalStorage<{
  userId: string;
  workspaceId: string | null;
  isSuperAdmin: boolean;
}>();
export function getCurrentRequestContext() {
  return requestContextStore.getStore() ?? null;
}
```

3. W middleware chain (w `app.ts` po requireAuth + requireWorkspace):
```typescript
app.use((req, res, next) => {
  if (!req.auth) return next();
  requestContextStore.run({
    userId: req.auth.sub,
    workspaceId: req.workspaceId ?? null,
    isSuperAdmin: req.auth.isSuperAdmin ?? false,
  }, () => next());
});
```

**Ale problem:** requireAuth i requireWorkspace są DZIŚ per-router, nie globalne. Router `/api/v2/public` nie ma tego middleware. Niewłaściwe wywołanie `requestContextStore.run` w niewłaściwym miejscu spowoduje że publiczne endpointy nie mają kontekstu — co po włączeniu NOBYPASSRLS zablokuje `public.routes.ts` (ale ta route już będzie używać prismaBg z Etapu 3, więc OK).

**Action item dla Etap 4:** dodać middleware kontekstu w każdym router pod `/api/v2/*` które są po auth — alternatywnie zrestrukturyzować chain.

**Weryfikacja:**
- Middleware ustawia kontekst — logowanie request → SELECT na Ticket z user ws_A → query zwraca tylko tickets z ws_A
- Trudne do zweryfikowania DOPÓKI user ma BYPASSRLS — kontekst się ustawia ale policies nie enforce'ują

**Rollback:** git revert + restart.

**Ryzyko:** średnie. Extension może mieć edge cases (np. raw queries, $queryRaw), do przetestowania lokalnie przed deploy.

**⏸️ Pauza: extension jest sporym refactorem. Do akceptacji przed kodowaniem.**
